# Smart Campus - Sensor & Room Management REST API

University of Westminster **5COSC022W.2 Client-Server Architectures** coursework project.

This is a Maven **JAX-RS** REST API implemented with **Jersey**. It uses in-memory Java collections only and is designed for NetBeans, Tomcat, GitHub submission, Postman demonstration, and README-based coursework report answers.

## Important Submission Note

Only source code and project configuration files should be committed to GitHub.

Do commit:

- `pom.xml`
- `README.md`
- `.gitignore`
- `nb-configuration.xml`
- `src/main/...` source files and web configuration

Do not commit generated output:

- `target/`
- `.class` files
- generated `.war` files
- IDE private folders
- logs and temporary files

The `.gitignore` file is included to keep Maven and NetBeans build artifacts out of the repository.

## Coursework Overview

The API manages:

- rooms on a smart campus
- sensors assigned to rooms
- historical readings for each sensor

The API root is:

```text
/api/v1
```

The implementation includes:

- discovery endpoint
- room management
- sensor management
- optional sensor filtering with `@QueryParam`
- nested readings using the Sub-Resource Locator pattern
- custom exception mappers
- request and response logging filters
- consistent JSON responses

## Run Instructions

### NetBeans and Tomcat

1. Open NetBeans.
2. Select **File > Open Project**.
3. Open this Maven project folder.
4. Configure Apache Tomcat 9 if NetBeans asks for a server.
5. Right-click the project and select **Clean and Build**.
6. Right-click the project and select **Run**.

Base URL:

```text
http://localhost:8080/smart-campus-api/api/v1
```

### Manual Tomcat Deployment

1. Run `mvn clean package`.
2. Copy `target/smart-campus-api.war` to Tomcat's `webapps` folder.
3. Start Tomcat.
4. Open `http://localhost:8080/smart-campus-api/api/v1`.

### NetBeans / Tomcat Troubleshooting

If NetBeans logs mention another application, such as `StockTicker` or a path like `CSAcoursework`, Tomcat is running an older deployment. Stop Tomcat, undeploy old applications from **Services > Servers > Apache Tomcat > Web Applications**, then run this project again.

If Tomcat reports a missing jar under `target/smart-campus-api/WEB-INF/lib`, stop Tomcat before running `mvn clean package`. That error usually means Maven cleaned the generated deployment while Tomcat was still undeploying it.

## Response Format

All normal success responses are JSON objects. The API does not return plain success strings or bare arrays.

Single-resource and action responses use this style:

```json
{
  "success": true,
  "message": "Room created successfully.",
  "data": {
    "id": "R-202",
    "name": "Seminar Room 202",
    "capacity": 40,
    "sensorIds": []
  },
  "path": "/smart-campus-api/api/v1/rooms",
  "timestamp": 1710000000000
}
```

Collection responses use this style:

```json
{
  "success": true,
  "message": "Rooms retrieved successfully.",
  "count": 2,
  "data": [
    {
      "id": "LAB-2",
      "name": "Computer Lab 2",
      "capacity": 35,
      "sensorIds": []
    }
  ],
  "path": "/smart-campus-api/api/v1/rooms",
  "timestamp": 1710000000000
}
```

Error responses use one consistent shape:

```json
{
  "timestamp": 1710000000000,
  "status": 422,
  "error": "Unprocessable Entity",
  "message": "Sensor cannot be created because roomId 'ROOM-DOES-NOT-EXIST' does not exist.",
  "path": "/smart-campus-api/api/v1/sensors"
}
```

Unexpected server errors are logged on the server and returned to the client as safe JSON. Raw Java stack traces are not sent to clients.

| Method | Endpoint | Description | Success |

| GET | `/api/v1/rooms` | List rooms with wrapper and count | 200 |
| POST | `/api/v1/rooms` | Create room | 201 |
| GET | `/api/v1/rooms/{roomId}` | Get room by id | 200 |
| DELETE | `/api/v1/rooms/{roomId}` | Delete empty room | 200 |
| GET | `/api/v1/sensors` | List sensors with wrapper and count | 200 |
| GET | `/api/v1/sensors?type=CO2` | Filter sensors by type | 200 |
| POST | `/api/v1/sensors` | Create sensor linked to room | 201 |
| GET | `/api/v1/sensors/{sensorId}` | Get sensor by id | 200 |
| GET | `/api/v1/sensors/{sensorId}/readings` | List sensor readings with wrapper and count | 200 |
| POST | `/api/v1/sensors/{sensorId}/readings` | Add reading and update current value | 201 |

## Error Status Codes

| Scenario | Status |
| --- | --- |
| Missing direct resource | 404 |
| Invalid JSON body or invalid field | 400 |
| Duplicate room or sensor id | 409 |
| Delete room that still has sensors | 409 |
| Create sensor with missing linked room id | 422 |
| Post reading to MAINTENANCE sensor | 403 |
| Wrong POST `Content-Type` | 415 |
| Unexpected server error | 500 |

## Sample JSON Bodies

Create room:

```json
{
  "id": "R-202",
  "name": "Seminar Room 202",
  "capacity": 40
}
```

Create active sensor:

```json
{
  "id": "S-CO2-202",
  "type": "CO2",
  "status": "ACTIVE",
  "currentValue": 410.5,
  "roomId": "R-202"
}
```

Create invalid linked sensor for `422` demo:

```json
{
  "id": "S-BAD",
  "type": "CO2",
  "status": "ACTIVE",
  "currentValue": 400,
  "roomId": "ROOM-DOES-NOT-EXIST"
}
```

Create sensor reading:

```json
{
  "value": 425.75
}
```

Optional reading timestamp:

```json
{
  "timestamp": 1710000000000,
  "value": 425.75
}
```

If `id` is missing for a reading, the API generates one. If `timestamp` is `0` or missing, the API uses the current server time. Reading values must be finite numbers, and timestamps cannot be negative or in the future.

## Coursework Question Answers

### Part 1

**JAX-RS resource lifecycle, request scope, shared data, and thread safety**

JAX-RS resource classes are normally request scoped, meaning a new resource instance may be created to handle an incoming request. Therefore, API data should not be stored in ordinary resource instance fields. This project keeps shared data in `InMemoryDataStore`, which is a singleton using `ConcurrentHashMap` and `CopyOnWriteArrayList`. Service methods synchronize multi-step operations such as creating a sensor and linking it to a room, reducing race-condition risk while keeping the design simple.

**HATEOAS and discovery**

HATEOAS means clients can discover useful links from API responses. The discovery endpoint returns links for rooms, sensors, and readings. This helps client developers and Postman users navigate the API without relying only on static documentation.

### Part 2

**Returning room IDs versus full room objects**

Returning only IDs saves bandwidth for very large collections, but clients then need extra requests to fetch useful details. Returning full room objects uses more bandwidth but reduces client-side work and is clearer for a coursework demonstration. This API returns full room objects inside a structured `data` field because the in-memory dataset is small and the response is easy to understand in Postman.

**DELETE idempotency**

DELETE is idempotent because repeating the same delete request leaves the system in the same final state: the room is absent. In this implementation, deleting an existing empty room returns success. Repeating the request returns `404 Not Found` because the room no longer exists, but the final server state remains unchanged.

### Part 3

**`@Consumes` mismatch and `415`**

POST methods use `@Consumes(MediaType.APPLICATION_JSON)`. If a client sends `text/plain`, XML, or another unsupported media type, Jersey rejects the request with `415 Unsupported Media Type`. The `WebApplicationExceptionMapper` formats that response as JSON so Postman does not show a default HTML page.

**Query parameters versus path parameters**

Path parameters identify a specific resource, such as `/rooms/R-202`. Query parameters filter or modify a collection result, such as `/sensors?type=CO2`. Filtering by sensor type is optional and may return many sensors, so `@QueryParam` is the better REST design.

**Why `422` is better than `404` for missing linked `roomId`**

When creating a sensor, the `/sensors` endpoint exists and the JSON request can be valid, but the `roomId` inside the payload may refer to a missing room. `422 Unprocessable Entity` is more precise because the problem is the meaning of the submitted entity, not the existence of the endpoint.

### Part 4

**Sub-Resource Locator pattern**

`SensorResource` has a locator method for `/{sensorId}/readings` that returns a dedicated `SensorReadingResource`. This keeps sensor logic separate from reading-history logic. It also avoids placing every nested path in one large resource class, which improves maintainability as the API grows.

**Historical readings and `currentValue`**

Each sensor has a reading history list. When a reading is posted successfully, it is appended to the history and the parent sensor's `currentValue` is updated. This keeps `GET /sensors/{sensorId}` consistent with the most recent reading.

### Part 5

**Custom exception mapping**

The API maps exceptions to structured JSON responses:

- `RoomNotEmptyException` -> `409 Conflict`
- `ResourceConflictException` -> `409 Conflict`
- `LinkedResourceNotFoundException` -> `422 Unprocessable Entity`
- `SensorUnavailableException` -> `403 Forbidden`
- `WebApplicationException` -> JSON versions of JAX-RS errors such as `400`, `404`, and `415`
- `Throwable` -> generic `500 Internal Server Error`

**Cybersecurity risk of stack traces**

Stack traces can reveal class names, package names, framework versions, filesystem paths, and internal implementation details. Attackers can use that information to target known vulnerabilities. This API logs unexpected errors on the server but returns a safe generic JSON message to clients.

**Why filters are better than logging inside every resource method**

JAX-RS filters centralize cross-cutting behavior. The `LoggingFilter` applies request and response logging to every endpoint automatically. This avoids duplicated `Logger.info()` calls in each resource method and keeps resource classes focused on API behavior.
